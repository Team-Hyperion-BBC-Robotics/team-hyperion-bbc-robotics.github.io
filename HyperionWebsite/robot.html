Software · Orientation & Stability

BNO055 IMU System

An Overview

In the heat of a match, the robot is constantly bumped, spun, and pushed by other players. Without a sense of direction, it would quickly become disoriented, driving toward its own goal by mistake. The BNO055 Inertial Measurement Unit (IMU) acts as the robot's "Inner Ear," providing a rock-solid sense of balance and heading.

Maintaining North

While the TSSP "eyes" find the ball, the IMU keeps the robot facing the correct way. We use the IMUPLUS mode, which combines accelerometer and gyroscope data to give us a stable "Heading" (the 0 to 360-degree angle the robot is facing).

  • Compass Reset: When we flick the calibration switch, the code resets the sensor's orientation. This defines "Front" for the rest of the game.
  • Drift Compensation: Standard gyroscopes "drift" over time, but the BNO055 uses internal algorithms to keep the heading precise even during high-speed collisions.
  • Active Correction: No matter which way the robot is told to drive, it is always trying to fight the rotation of the wheels to stay facing the opponent's goal.

The Stability Math

To keep the robot straight, we use a PID Controller (Proportional, Integral, Derivative). The IMU tells us our current angle, and we compare it to our target (usually 0°). The difference between them is the "Error." The robot then applies a counter-rotation (Correction) to cancel out that error.

// 1. Get the current bearing from the IMU bno.getEvent(&bearing); float currentHeading = bearing.orientation.x; // 2. Normalise the angle to [-180, 180] for easier math float error = com.normaliseAngle180(currentHeading); // 3. Calculate Correction (_cor) using PID _cor = -correction.update(error, 0.0); // 4. Apply to Motors motors.run(_spd, _dir, _cor);

Software · Ball Detection

TSSP System

An Overview

Imagine our robot is standing in the middle of a dark room. Somewhere in that room, there is a glowing orange ball. Our goal is to find that ball and kick it into a goal. We use two electronic "brains" (called Teensys) to do this. One brain acts like the Eyes, and the other acts like the Captain.

The "Secondary" Teensy

This part of the code is responsible for looking around. We have several sensors (TSSP sensors) placed in a circle around the robot. These sensors are like little eyes that can only see infrared light from the ball.

  • The infrared ball blinks very fast. The "Eyes" look at each sensor 255 times in a row. Every time a sensor sees a flash of light, it adds a point.
  • If the sensor on the front-right has 200 points and the one on the back has 0, the robot knows the ball is somewhere to the front-right.
  • The code takes the top four strongest sensor readings and combines them using vector summation to calculate both the angle and strength of the ball signal.

The "Eyes" Math

The system identifies the ball's position by isolating the four highest sensor readings. Each sensor is treated as a vector, where the physical orientation of the sensor represents the angle and the point count represents the magnitude. By plotting these on a Cartesian plane and performing a vector summation, the robot calculates a single resultant vector.

// TSSP Update — Ball Direction and Strength Calculation for (i = 0; i < 255; i++) { tsspValues[j] += 1 - digitalRead(tsspPins[j]); } x = sum(tsspSortedValues * cos(sensorAngle)); y = sum(tsspSortedValues * sin(sensorAngle)); ballStr = ((3 * top[0]) + (2 * top[1]) + top[2] + top[3]) / 7.0; if (ballStr != 0) { ballDir = 360 - fmod((RAD_TO_DEG * atan2f(y, x)) - 90, 360); } else { ballDir = 0; }

The "Captain" (Primary's) Math

The Captain's brain uses exponential functions to make decisions. Instead of a simple turn, it calculates a Strength Factor based on the ball's signal intensity. This factor determines the Angle Shift — the mathematical curve required to orbit the ball.

// 1. Normalize ball direction to [-180, 180] float relativeDir = (bInfo.dir() > 180) ? (bInfo.dir() - 360) : bInfo.dir(); // 2. Calculate Strength Factor float ratio = constrain(bInfo.str() / ORBIT_TUNER, 0.0, 1.0); float strengthFactor = constrain(0.02 * ratio * expf(4.5 * ratio), 0.0, 1.0); // 3. Calculate Angle Shift (Orbit Offset) float angleEffect = min(0.4 * expf(0.25 * abs(relativeDir)) - 0.4, 90.0); float totalOffset = strengthFactor * angleEffect; // 4. Apply direction adjustment and scale speed float adjustment = (relativeDir < 0) ? -totalOffset : totalOffset; mInfo.dir = fmod(bInfo.dir() + adjustment, 360.0); mInfo.spd = BASE_SPEED + (SURGE_SPEED - BASE_SPEED) * (1.0 - totalOffset / 90.0);

Software · Line Avoidance

LIGHT SYSTEM

The LightSystem library detects the white boundary lines of the RoboCup field via an underside IR phototransistor array. Each sensor emits IR light downward and measures the reflected intensity — the white field lines reflect significantly more than the dark playing surface, producing a clear contrast spike that the library converts into a line direction and state value. In the state selection priority chain, STATE_LINE_AVOID sits immediately after calibration — meaning it pre-empts both attack and defend behaviour the instant any line is detected.

Avoidance Logic

When ls.get_line_dir() returns anything other than -1, the robot is on the line. The avoidance direction is computed as (line_dir + 180°) mod 360 — the exact opposite of where the line was detected.

Vertical Correction in Defend

The light system also plays a role in the defender's vector sub-state outside of full line avoidance mode. The get_line_state() value feeds a vertical PID (vert) that gently pushes the defender away from the back wall.

STATE_LINE_AVOID in loop() _spd = -lineAvoid.update(ls.get_line_state(), 0.0); _dir = fmod(ls.get_line_dir() + 180.0, 360); In LWDEFEND_VECTOR — soft wall pushback float vertVect = vert.update(ls.get_line_state(), 1.6);

Software · Vision

OPENMV CAMERA

The Camera library wraps the OpenMV RT's UART output into structured data. The OpenMV runs a MicroPython colour blob tracking script on its own dedicated processor — detecting yellow (attack goal) and blue (defend goal) blobs in the camera frame, computing the centroid angle relative to the camera's horizontal centre, and transmitting that angle and blob size to the Teensy over a dedicated UART link every frame.

Goal Tracking Switch

cam.update() takes the state of the GOAL_TRACK_SWITCH as a parameter — a physical DIP switch on the robot. This allows the team to instantly disable goal tracking at competition time if the camera is producing unreliable readings in a given lighting environment.

Field Localisation

The camera also provides a cam.robot() object for field position estimation. When the ball is completely lost (TSSP strength zero), cam.localise_to(x, y) computes a movement vector driving the robot toward a target field coordinate.

Software · Inter-Robot Communication

BLUETOOTH COMMUNICATION

An Overview

Imagine we have two robots on a soccer field. If both robots chase the ball at the same time, they will crash into each other or leave the goal wide open. To play as a team, they need to talk. We use HC-05 Bluetooth modules to let the robots "whisper" to each other across the field.

How they Talk (The "Packet" System)

The robots don't just send random numbers; they send a structured "envelope" of information called a Packet. Every message starts with two Start Bytes (the number 255 twice) to ensure valid data. The packet contains: Ball Strength, Ball Direction, Goal Angle, Goal Distance, Battery Level, and X/Y coordinates.

How they Decide (The "Role Weighting" Logic)

Each robot calculates a "Score" for itself and its teammate based on several factors. The robot with the higher score becomes the Attacker. Factors include Ball Strength (0-40pts), Attack Cone (0 or 20pts), Goal Alignment (0 or 15pts), Defensive Gap (0-15pts), and Battery Level (0-10pts).

Fail-Safe's

In the heat of a game, things go wrong. Dead Radio: if Bluetooth disconnects, the robot immediately defaults to Defender. Blind spots: if neither robot can see the ball, they stop switching roles. Battery/Fuel Rule: if a robot's battery is critically low, it is forced into Defender role.

int Bluetooth::roleWeighting(const SendData& r) { // 1. Give points for being close to the ball int score = r.ballStr * 0.4f; // 2. Bonus if ball is right in front score += (r.ballDir <= 30 || r.ballDir >= 330) ? 20 : 0; // 3. Bonus if we are facing the goal we want to hit score += (r.goalAng <= 25 || r.goalAng >= 335) ? 15 : 0; return score; }
